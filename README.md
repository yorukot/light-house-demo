# AI 勞動條款風險審查 Demo 規劃

這是一個靜態網頁 demo，用來呈現「勞動契約附件：個人資料蒐集、AI 訓練與數位行為監控條款」被輸入後，系統如何產生指標式風險評級與內容分析結果。這個版本只做展示情境，不提供正式法律意見，也不連接後端或真實 AI 模型。

## 1. Demo 目標

本 demo 的目標是把一份公司員工監控與 AI 訓練條款，轉成可被法遵、HR、工會或研究者快速閱讀的審查輸出。

核心使用情境：

1. 使用者將「勞動契約附件」貼入左側輸入區。
2. 使用者點擊分析。
3. 介面模擬 AI 正在拆解條款，例如辨識資料蒐集行為、比對蒐集目的、檢查比例原則、檢查拒絕權與程序保障。
4. 右側輸出一份表格，包含「指標」「評級」「內容分析與偵測結果」。
5. 表格之外呈現整體風險、主要疑慮與建議補強項目，方便 demo 時快速說明。

## 2. 輸入與輸出定義

### 輸入

輸入是一份契約條款文字，範例包含五個段落：

- 數位行為監控與蒐集
- 人工智慧技術與資料利用
- 跨境傳輸與資料存取
- 保存期限與權利行使
- 拒絕蒐集之處理

系統會以條款文字中的關鍵概念作為展示用判斷來源，例如：

- 鍵盤、滑鼠、螢幕、應用程式、會議 metadata
- 去識別化、匿名化、AI 模型訓練、RAG、流程自動化
- 績效考核、升遷、職務調整、自動化決策、人工覆核
- DPO、跨境傳輸、SCCs、存取稽核日誌
- 拒絕權、不得解僱、不得減薪、不得歧視性處分

### 輸出

輸出是一個指標表格，欄位固定如下：

| 欄位 | 說明 |
| --- | --- |
| 指標 | 被檢查的法遵或倫理問題 |
| 評級 | 有、無、有違法、不違法但有疑慮、法制漏洞建議、無疑慮 |
| 內容分析與偵測結果 | 用自然語言說明系統為何給出該評級 |

目前 demo 預設輸出的指標為：

- 有無取得資料行為
- 勞方資料種類、範圍、時間
- 蒐集處理利用目的與手段
- 必要性與替代手段
- 已識別化資料的蒐集行為
- 利益權衡
- 勞方拒絕權
- 磋商、去識別化、安全措施

## 3. 評級邏輯設計

此版本採用前端 JavaScript 的展示用規則，不做真正 NLP 或法律推論。

### 規則方向

- 若文字包含「鍵盤」「滑鼠」「螢幕」「監控」「蒐集」，判定存在資料取得行為。
- 若文字包含大量監控項目但缺少資料最小化描述，標示「不違法但有疑慮」或「法制漏洞建議」。
- 若文字包含「AI」「模型訓練」「RAG」「自動化決策」，標示 AI 訓練與自動化決策用途已揭露，但仍需檢查具體流程。
- 若文字包含「拒絕權」「不得作為解僱」「減薪」「降職」「歧視性處分」，標示拒絕權較無疑慮。
- 若文字包含「去識別化」「匿名化」「SCCs」「DPO」「稽核日誌」，標示有安全措施，但若缺少程序上磋商或特種資料隔離，仍保留疑慮。

### 為什麼不直接標示合法或違法

使用者提供的法制架構仍有大量需要個案判斷的部分，例如資料是否真正必要、是否屬就業所需、勞工同意是否出於自由意志、公司是否能提出替代手段。demo 因此將輸出設計為「風險檢測」而非「法律結論」。

## 4. 使用者流程

1. 進入頁面後，左側已填入範例契約條款。
2. 使用者可以直接按「開始分析」，也可以修改文字再分析。
3. 分析開始後，右側出現進度列與思考紀錄。
4. 分析完成後，右側產生：
   - 整體評估：中度風險
   - 摘要：有明確資料蒐集行為、AI 訓練目的有揭露、拒絕權相對完整，但資料最小化、特種資料隔離與程序磋商不足
   - 指標表格：逐項列出評級與分析
5. 使用者可按重設回到範例條款。

## 5. 視覺與互動方向

設計語氣：現代公司內部治理工具，偏安靜、清楚、可掃讀，不走過度科技感或行銷頁風格。

### 視覺原則

- 第一個畫面直接是可用工具，不做 landing page。
- 使用左右雙欄：左側為契約輸入，右側為審查輸出。
- 色彩使用指定色盤：`#000000`、`#233D4D`、`#FE7F2D`、`#EAECF0`。
- 介面不使用圓角。
- 介面不使用漸層。
- 表格是主要輸出，不以裝飾性大圖搶焦點。
- 使用小型 canvas 視覺化「條款到風險指標」的掃描流程，作為 demo 的動態視覺資產。

### 互動原則

- 按下分析後不立即顯示全部結果，而是分步顯示 AI 思考狀態。
- AI 思考狀態不假裝是真實模型推理，只作為 demo 節奏與敘事。
- 表格完成後才顯示重點建議，讓 demo 流程有明確前後。
- 手機版改為上下排列，表格可水平捲動。

## 6. 檔案結構

```text
.
├── README.md
├── index.html
├── styles.css
└── script.js
```

### `index.html`

負責頁面結構：

- 應用程式標題列
- 契約輸入 textarea
- 分析與重設按鈕
- AI 思考區
- 整體風險摘要
- 指標輸出表格

### `styles.css`

負責視覺與 responsive layout：

- 雙欄工具布局
- 表格、狀態 badge、按鈕、textarea
- 進度列與 loading 狀態
- 手機版斷點

### `script.js`

負責 demo 邏輯：

- 填入範例契約文字
- 模擬分析步驟
- 根據文字產生展示用評級
- 渲染輸出表格與摘要
- canvas 掃描動畫

## 7. Demo 範圍限制

此版本刻意不處理：

- 真實自然語言理解
- 真實法律資料庫查詢
- 後端儲存
- 使用者帳號
- PDF 或 Word 匯入
- 多語言
- 實際 GDPR、EU AI Act 或台灣法規條文自動比對

若後續要進一步產品化，可以新增：

- 契約分段解析
- 條款引用來源標註
- 風險分數校準
- 使用者可調整的審查指標
- 匯出成 PDF 或 Word 報告
- 法條資料庫與判斷依據追蹤
- 工會或員工端的申訴與協商流程模板

## 8. 本地執行

這是純靜態 demo，直接打開 `index.html` 即可。

也可以使用任何簡單靜態伺服器，例如：

```bash
python3 -m http.server 8080
```

然後開啟：

```text
http://localhost:8080
```

## 9. GitHub Pages 部署

專案已包含 GitHub Actions workflow：

```text
.github/workflows/deploy.yml
```

部署方式：

1. 將專案推到 GitHub repo 的 `main` 分支。
2. 到 repo 的 Settings → Pages。
3. 將 Build and deployment 的 source 設為 GitHub Actions。
4. workflow 會把 `index.html`、`styles.css`、`script.js` 複製到 `_site`，再部署到 GitHub Pages。

如果 repo 的預設分支不是 `main`，需要修改 `.github/workflows/deploy.yml` 裡的 branch 設定。

## 10. Demo 交付標準

完成狀態應包含：

- 首頁就是可操作的契約審查 demo。

- 輸出表格包含使用者指定的三欄。
- 視覺呈現具有現代企業產品感，但不過度裝飾。
